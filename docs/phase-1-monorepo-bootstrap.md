# Фаза 1. Monorepo Bootstrap baseline

## Статус документа

Это рабочий baseline-документ для закрытия `Фазы 1`.

Документ фиксирует организационные решения, которые нужны до активной реализации домена и фич:

- структура монорепы;
- границы Python и Node toolchains;
- локальная инфраструктура;
- `env`-модель;
- единые команды разработчика;
- критерии готовности bootstrap-фазы.

Решения, уже подтвержденные как контекст:

- frontend workspace управляется через `bun workspaces`;
- единая оболочка команд делается через `Makefile`;
- стартовая локальная infra для `v1`: `postgres + mailpit + minio`.

## 1.1. Цель Фазы 1

Подготовить репозиторий так, чтобы следующие фазы шли без организационного шума.

После завершения этой фазы команда должна иметь:

- понятную структуру каталогов;
- отдельные `api` и `web` workspaces;
- shared package для API-типов;
- воспроизводимую локальную инфраструктуру;
- понятные переменные окружения;
- один очевидный способ запускать типовые dev-команды.

## 1.2. Границы новой кодовой базы

### Что считается новой системой

Новая кодовая база живет отдельно от `reference/`.

`reference/` остается:

- архивом старой реализации;
- источником требований;
- источником примеров данных и пользовательских сценариев.

`reference/` не используется:

- как место для нового кода;
- как слой совместимости;
- как промежуточная среда миграции.

### Практическое правило

Все новое развитие идет только в новой структуре монорепы.

## 1.3. Предлагаемая структура репозитория

```text
.
├── apps/
│   ├── api/
│   └── web/
├── packages/
│   └── api-types/
├── infra/
├── docs/
├── reference/
├── Makefile
├── README.md
└── .env.example
```

### Ответственность верхнеуровневых каталогов

#### `apps/api`

Python API-приложение.

Там живут:

- entrypoint приложения;
- конфиг;
- http layer;
- доменная модель;
- сервисы;
- репозитории;
- миграции;
- backend tests.

#### `apps/web`

SPA-приложение на `SvelteKit + TypeScript`.

Там живут:

- маршруты;
- layout;
- UI-компоненты;
- query layer;
- typed API client;
- frontend tests;
- storybook config.

#### `packages/api-types`

Shared package для типов, сгенерированных из OpenAPI.

Это не место для ручных бизнес-интерфейсов. Это generated-first пакет.

#### `infra`

Локальная инфраструктура и все, что нужно для bootstrap/dev runtime:

- `docker-compose`;
- volume-стратегия;
- env для инфраструктуры;
- возможно helper scripts, если они действительно нужны.

#### `docs`

Документация по фазам, решениям, архитектуре и процессу.

#### `reference`

Только чтение и анализ. Не участвует в новом runtime.

## 1.4. Toolchain boundaries

### Backend side

- отдельный Python toolchain;
- пакетный менеджер и workflow вокруг Python управляются отдельно от Node-части;
- backend не зависит от frontend toolchain для запуска.

### Frontend side

- frontend живет как `bun` workspace;
- `apps/web` и `packages/api-types` находятся в одном Node-контуре;
- все frontend-команды должны быть воспроизводимы через `bun`.

### Корневой уровень

Корень репозитория нужен не для смешивания toolchains, а для orchestration:

- `Makefile`;
- документация;
- общие entry-команды;
- общие env-примеры;
- запуск infra.

### Практическое правило

- Python-команды не прячутся внутрь Node-скриптов;
- Node-команды не прячутся внутрь Python-скриптов;
- orchestration идет через `Makefile`, а не через вложенную магию.

## 1.5. Baseline для `apps/api`

### Что должно появиться

Минимальный backend workspace с корректными зонами ответственности.

### Предлагаемая структура

```text
apps/api/
├── src/
│   └── app/
│       ├── api/
│       ├── config/
│       ├── domain/
│       ├── repositories/
│       ├── services/
│       └── main.py
├── tests/
├── migrations/
├── pyproject.toml
└── README.md
```

### Что важно зафиксировать сразу

- `src` layout;
- отдельные `tests`;
- место для миграций;
- понятный entrypoint;
- отсутствие свалки вида "все в одной папке app".

### Что не нужно делать в Фазе 1

- не нужно наполнять домен бизнес-логикой;
- не нужно сразу поднимать все зависимости production-уровня;
- не нужно проектировать весь API-контур до `Фазы 2`.

## 1.6. Baseline для `apps/web`

### Что должно появиться

Минимальный frontend workspace с правильным делением UI, маршрутов и API-слоя.

### Предлагаемая структура

```text
apps/web/
├── src/
│   ├── lib/
│   │   ├── api/
│   │   ├── query/
│   │   ├── stores/
│   │   └── ui/
│   └── routes/
├── e2e/
├── stories/
├── package.json
├── tsconfig.json
└── README.md
```

### Что важно зафиксировать сразу

- `routes` только для маршрутов и композиции экранов;
- `lib/ui` только для компонентов;
- `lib/api` и `lib/query` отделены от компонентов;
- `stores` не используются как замена server state;
- `e2e` и `stories` имеют отдельное место.

### Что не нужно делать в Фазе 1

- не нужно проектировать весь дизайн-системный слой;
- не нужно сразу наполнять SPA экранами продукта;
- не нужно вручную писать API types.

## 1.7. Baseline для `packages/api-types`

### Назначение

Связать backend OpenAPI и frontend TypeScript-типизацию.

### Принцип

Пакет считается generated-first.

Значит:

- основное содержимое пакета генерируется;
- ручные типы и самодельные копии DTO не приветствуются;
- фронтенд импортирует типы отсюда, а не из произвольных локальных файлов.

### Что должно появиться

- package manifest;
- каталог для generated output;
- понятная команда регенерации.

## 1.8. Локальная инфраструктура

### Состав инфраструктуры `v1`

- `postgres`
- `mailpit`
- `minio`

### Ответственность сервисов

#### `postgres`

- основная локальная БД;
- dev и test-like сценарии;
- база для миграций и локальной интеграции.

#### `mailpit`

- прием и просмотр исходящих писем локально;
- тестирование auth и notification flows.

#### `minio`

- локальное S3-совместимое хранилище;
- загрузка и просмотр медиа;
- возможность рано проверить file workflow.

### Что не добавляем на старте

- `redis`;
- фоновые воркеры;
- reverse proxy;
- observability stack;
- production-like orchestration.

Если позже понадобится, это добавляется отдельно и осознанно.

### Требование к infra

Локальная infra должна подниматься одной командой и не требовать ручной установки этих сервисов на машину.

## 1.9. Env-модель

### Общие правила

- все обязательные переменные описываются в `.env.example`;
- имена должны быть достаточно явными, чтобы не читать код ради их смысла;
- секреты в репозиторий не коммитятся;
- dev-конфиг должен быть воспроизводим локально с минимальной ручной правкой.

### Группы переменных

#### API

Примеры:

- app env;
- app host/url;
- database url;
- session/auth secret;
- mail config;
- storage config;
- CORS/origin config, если потребуется.

#### Web

Примеры:

- public app url;
- api base url;
- feature flags, если появятся;
- публичные настройки, допустимые для браузера.

#### Infra

Примеры:

- postgres db/user/password;
- minio access key/secret key;
- mailpit ports;
- локальные volume-related настройки при необходимости.

### Практический принцип

Если переменную нельзя понять по имени, naming надо переделать.

## 1.10. Команды разработчика

### Решение

Единая оболочка команд строится через `Makefile`.

### Что должны покрывать команды верхнего уровня

- поднять infra;
- остановить infra;
- запустить api;
- запустить web;
- запустить оба приложения;
- прогнать backend lint;
- прогнать frontend lint;
- прогнать tests;
- сгенерировать API types.

### Требование к командам

- короткие;
- предсказуемые;
- без необходимости помнить длинные tool-specific invocation details.

### Практическое правило

Для типовых действий у разработчика должен быть один очевидный путь.

## 1.11. Onboarding и smoke-проверка

### Что должен уметь сделать новый разработчик

1. Склонировать репозиторий.
2. Скопировать `.env.example`.
3. Поднять локальную infra.
4. Запустить API.
5. Запустить web.
6. Понять, где лежат основные workspaces.
7. Запустить хотя бы базовую проверку качества.

### Что нужно задокументировать

- prerequisites;
- последовательность bootstrap-шагов;
- базовые команды;
- где искать env;
- где лежит новый код, а где архив старой версии.

### Минимальный smoke check

Считаем bootstrap рабочим, если локально:

- infra стартует;
- API стартует;
- web стартует;
- обе части не требуют ручного ковыряния после стандартного bootstrap.

## 1.12. Результат Фазы 1

На выходе `Фазы 1` должно существовать:

- новая структура монорепы;
- `apps/api`;
- `apps/web`;
- `packages/api-types`;
- `infra` с локальными сервисами;
- `.env.example`;
- `Makefile`;
- краткая инструкция запуска;
- воспроизводимый bootstrap-path для нового разработчика.

## 1.13. Definition of Done

`Фаза 1` считается закрытой, если:

- структура репозитория утверждена и реализована;
- новый код отделен от `reference/`;
- `bun` workspace для frontend-части настроен;
- `Makefile` покрывает типовые команды;
- `postgres`, `mailpit` и `minio` поднимаются локально;
- переменные окружения описаны и понятны;
- `api` и `web` запускаются независимо;
- onboarding-путь проверен хотя бы одним полным локальным прогоном.

## Что это дает следующей фазе

После этого можно переходить к `Фазе 2` без возврата к вопросам:

- как разложить репозиторий;
- где хранить shared types;
- чем запускать локальную infra;
- как договориться о dev-командах;
- где заканчивается frontend и начинается backend.
