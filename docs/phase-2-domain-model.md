# Фаза 2. Domain Model baseline

## Статус документа

Это опорный baseline-документ для закрытия `Фазы 2`.

Его задача:

- зафиксировать доменные сущности `v1`;
- определить их ответственность;
- зафиксировать связи, ключевые поля и статусы;
- описать permission rules;
- определить файловую модель;
- подготовить базу для `Фазы 3`: миграций, сервисов и API DTO.

Документ опирается на уже подтвержденные решения из `Фазы 0` и `Фазы 1`.

## 2.1. Цель доменного слоя

Доменный слой должен описывать не экраны и не таблицы “как получится”, а устойчивую модель предметной области.

Для `v1` нам нужна модель, которая уверенно покрывает:

- каталог объявлений;
- личные объявления пользователя;
- изображения;
- комментарии;
- staff-модерацию категорий и комментариев.

## 2.2. Набор сущностей `v1`

### Обязательные сущности

- `User`
- `Category`
- `Ad`
- `AdImage`
- `Comment`

### Необязательные сущности

Эти сущности не включаем в обязательный `v1`, если не всплывет реальная необходимость:

- `ModerationAction`
- `AuditLog`
- `DraftAd`
- `Favorite`

### Решение

Для `v1` доменную модель держим компактной. Новая сущность появляется только тогда, когда без нее нельзя выразить бизнес-правило или сценарий.

## 2.3. Ответственность сущностей

### `User`

Отвечает за:

- учетную запись;
- доступ в систему;
- владение объявлениями;
- участие в комментариях;
- роль пользователя.

`User` не должен превращаться в склад всего подряд. В `v1` это прежде всего identity + ownership.

### `Category`

Отвечает за:

- структуру каталога;
- группировку объявлений;
- иерархию категорий.

В модели допускаем иерархию через `parent_id`, но UI `v1` ориентируем на максимум два уровня.

### `Ad`

Это центральная сущность продукта.

Отвечает за:

- объявление как единицу контента;
- принадлежность пользователю;
- привязку к категории;
- публичную видимость;
- базовую коммерческую информацию.

### `AdImage`

Отвечает за:

- изображения объявления;
- порядок изображений;
- хранение метаданных, связанных с файлом;
- привязку к объявлению.

Это отдельная сущность, а не просто массив строк в `Ad`.

### `Comment`

Отвечает за:

- текст комментария;
- связь с объявлением;
- связь с автором;
- модерационный статус.

В `v1` комментарии оставляют только авторизованные пользователи.

## 2.4. Связи между сущностями

### `User -> Ad`

- один пользователь может иметь много объявлений;
- одно объявление принадлежит ровно одному пользователю.

### `Category -> Category`

- категория может иметь родителя;
- корневая категория родителя не имеет;
- категория может иметь много дочерних категорий.

### `Category -> Ad`

- одна категория содержит много объявлений;
- одно объявление относится к одной категории.

### `Ad -> AdImage`

- одно объявление может иметь много изображений;
- одно изображение принадлежит ровно одному объявлению.

### `Ad -> Comment`

- одно объявление может иметь много комментариев;
- один комментарий относится ровно к одному объявлению.

### `User -> Comment`

- один пользователь может оставить много комментариев;
- один комментарий принадлежит ровно одному пользователю.

## 2.5. Предлагаемая ER-логика

В терминах модели:

- `User` владеет `Ad`;
- `Ad` принадлежит `Category`;
- `Ad` агрегирует `AdImage`;
- `Ad` агрегирует `Comment`;
- `Comment` создается `User`;
- `Category` образует дерево через self-reference.

## 2.6. Ключевые поля сущностей

Ниже не окончательная SQLAlchemy-модель, а baseline для проектирования первой схемы БД.

### `User`

Минимально:

- `id`
- `email`
- `username` или `display_name`
- `password_hash`
- `role`
- `is_active`
- `created_at`
- `updated_at`

Дополнительно стоит решить при реализации:

- нужен ли отдельный `full_name`;
- нужен ли `email_verified_at` на будущее, даже если email verification не входит в `v1`.

### `Category`

Минимально:

- `id`
- `name`
- `slug`
- `parent_id`
- `sort_order`
- `is_active`

### `Ad`

Минимально:

- `id`
- `owner_id`
- `category_id`
- `title`
- `description`
- `price`
- `contacts`
- `status`
- `deleted_at`
- `created_at`
- `updated_at`

### `AdImage`

Минимально:

- `id`
- `ad_id`
- `storage_key`
- `original_filename`
- `content_type`
- `size_bytes`
- `sort_order`
- `is_primary`
- `created_at`

### `Comment`

Минимально:

- `id`
- `ad_id`
- `author_id`
- `content`
- `status`
- `created_at`
- `updated_at`

## 2.7. Решения по типам данных

### Цена

Цена хранится как денежный тип, не как `float`.

Практическое решение для БД:

- `NUMERIC/DECIMAL`

### Валюта

Для `v1` отдельное поле `currency` не включаем в обязательную модель.

Рабочее допущение:

- в `v1` проект живет в одной базовой валюте.

Если мультивалютность реально понадобится позже, это можно расширить отдельной миграцией без ломки ядра текущей модели.

### Иерархия категорий

Категории моделируются через:

- `parent_id nullable`

Это дает гибкость модели без усложнения `v1`-UI.

### Автор комментария

Комментарий в `v1` связан с реальным `User`.

Значит:

- не используем строковое поле автора как источник истины;
- имя автора выводим из пользователя или из отдельного snapshot-поля, если позже появится такая необходимость.

### Владелец объявления

У объявления всегда есть явный owner.

Никаких скрытых form fields как в старой версии.

## 2.8. Lifecycle и статусы

### Статусы пользователя

Для `v1` достаточно:

- `active`
- `disabled`

Если отдельная верификация email позже вернется, это не обязано быть статусом пользователя. Это может быть отдельный атрибут.

### Статусы объявления

Для `v1` предлагаю:

- `active`
- `hidden`

Где:

- `active` означает, что объявление видно в публичном каталоге;
- `hidden` означает, что объявление скрыто владельцем или staff.

Решение:

- не вводить отдельный `draft` в `v1`;
- не вводить сложную премодерационную воронку.

### Soft delete объявления

Для `v1` удаление объявления делаем через `soft delete`.

Технически это выражается через:

- `deleted_at nullable`

С точки зрения поведения системы это означает дополнительное эффективное состояние:

- `deleted`

Но не обязательно как отдельное значение `status`.

### Статусы комментария

Для `v1` предлагаю:

- `published`
- `hidden`

Где:

- `published` виден публично;
- `hidden` скрыт staff-ом.

## 2.9. Допустимые переходы состояний

### `Ad`

- `active -> hidden`
- `hidden -> active`
- `active -> deleted`
- `hidden -> deleted`

В `v1` восстановление soft-deleted объявления не считаем обязательным пользовательским сценарием.

### `Comment`

- `published -> hidden`
- `hidden -> published`, если staff разрешит восстановление

### `User`

- `active -> disabled`
- `disabled -> active`

## 2.10. Доменные инварианты

Ниже правила, которые должны всегда оставаться истинными.

### Для `User`

- email уникален;
- роль пользователя валидна;
- отключенный пользователь не должен считаться полноценным активным actor-ом системы.

### Для `Category`

- категория не может ссылаться на саму себя как на родителя;
- циклы в дереве категорий запрещены;
- имя/slug должны быть пригодны для публичного каталога.

### Для `Ad`

- у объявления всегда есть владелец;
- у объявления всегда есть категория;
- цена не может быть отрицательной;
- только владелец или staff может менять объявление;
- скрытое объявление не попадает в публичный каталог.
- soft-deleted объявление не попадает в публичный каталог;
- soft-deleted объявление не должно участвовать в обычном пользовательском CRUD как активный объект.

### Для `AdImage`

- изображение всегда привязано к объявлению;
- порядок изображений должен быть детерминирован;
- максимум одно изображение может быть primary, если используем этот флаг;
- удаление объявления не должно оставлять подвешенные файловые записи.

### Для `Comment`

- комментарий всегда относится к одному объявлению;
- комментарий всегда имеет автора;
- только staff может скрыть чужой комментарий;
- скрытый комментарий не показывается публично.

## 2.11. Permission model на уровне домена

### `guest`

Может:

- смотреть каталог;
- смотреть карточки объявлений;
- искать и фильтровать.

Не может:

- создавать объявления;
- комментировать;
- редактировать что-либо.

### `user`

Может:

- создавать объявления;
- редактировать свои объявления;
- удалять свои объявления;
- загружать изображения к своим объявлениям;
- публиковать комментарии;
- управлять своим профилем.

Не может:

- редактировать чужие объявления;
- модерировать комментарии;
- управлять категориями.

### `staff`

Может:

- делать все, что может `user`;
- скрывать комментарии;
- управлять категориями;
- скрывать объявления при необходимости;
- выполнять служебные действия в backoffice.

## 2.12. Файловая модель

### Общий принцип

Файл не живет как "URL в строке". У нас есть:

- объект в S3-совместимом хранилище;
- запись в БД с метаданными и связью с объявлением.

### Source of truth

Source of truth для хранения файла:

- `storage_key`

А не внешний URL.

Почему:

- URL можно пересобрать;
- storage key стабилен внутри нашей модели хранения;
- это лучше переносится между dev/stage/prod.

### Что храним в `AdImage`

- `storage_key`
- `original_filename`
- `content_type`
- `size_bytes`
- `sort_order`
- `is_primary`

### Что делаем в `v1`

- оригиналы храним в object storage;
- список изображений упорядочиваем через `sort_order`;
- главное изображение можно задавать через `is_primary`.

### Что не делаем в `v1`

- сложный media pipeline;
- асинхронную генерацию производных форматов;
- несколько медиа-типов кроме изображений.

### Открытый вопрос на реализацию

Превью можно:

- либо генерировать на лету/через фронт только из оригинала на старте;
- либо отложить отдельную систему thumbnail generation до момента, когда она реально понадобится.

Для `v1` это не блокер доменной модели.

## 2.13. Граница между доменом и API DTO

### Принцип

Внутренние модели и внешние API DTO не обязаны совпадать один в один.

Это важно, потому что:

- домен описывает реальность системы;
- API описывает конкретное внешнее представление для клиента.

### Какие представления точно понадобятся

#### Для объявлений

- `AdSummary`
- `AdDetail`
- `CreateAdPayload`
- `UpdateAdPayload`

#### Для категорий

- `CategoryTreeNode`
- возможно отдельное staff representation позже

#### Для комментариев

- `CommentView`
- `CreateCommentPayload`

#### Для пользователя

- `CurrentUserView`
- `RegisterPayload`
- `LoginPayload`

### Практический вывод

Схема БД не должна автоматически становиться схемой публичного API.

## 2.14. Что должно стать артефактом после Фазы 2

На выходе этой фазы у нас должны быть:

- список сущностей `v1`;
- список ключевых полей;
- схема связей;
- список статусов и переходов;
- список инвариантов;
- permission rules;
- файловая модель;
- черновой список DTO-представлений.

## 2.15. Definition of Done

`Фаза 2` считается закрытой, если:

- по каждой сущности понятно, зачем она нужна;
- связи между сущностями не оставляют двусмысленности;
- цена, ownership, комментарии и категории зафиксированы на уровне модели;
- lifecycle ключевых сущностей описан;
- permission model зафиксирована;
- файловая модель определена;
- можно переходить к миграциям и service layer без повторного domain discovery.

## Что вам стоит особенно проверить

При чтении этого документа стоит особенно проверить:

- устраивает ли набор сущностей `v1`;
- достаточно ли статусов `active/hidden` при наличии `soft delete` через `deleted_at`;
- устраивает ли модель комментариев только для авторизованных пользователей;
- устраивает ли отказ от отдельного `currency` в `v1`;
- нужен ли `is_primary` у изображения в `v1`;
- устраивает ли `soft delete` как модель удаления объявления;
- устраивает ли объем полномочий `staff`.
