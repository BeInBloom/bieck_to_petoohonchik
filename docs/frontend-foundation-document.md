# Frontend Foundation Document

## Цель

Зафиксировать базовый дизайн-контур для `apps/web`, чтобы можно было последовательно реализовывать:

- auth flow;
- список объявлений;
- детальную страницу объявления;
- базовую навигацию по категориям и пользовательским действиям.

Документ опирается на legacy-проект только как на источник:

- пользовательских сценариев;
- состава экранов;
- информационной архитектуры;
- доменной терминологии.

Документ не предполагает перенос:

- legacy-визуального стиля;
- Bootstrap/Django-шаблонной структуры;
- Angular-архитектуры;
- устаревших UI-паттернов, не нужных для нового продукта.

## Источники референса

При подготовке документа были просмотрены:

- `reference/bboard/main/templates/layout/basic.html`
- `reference/bboard/main/templates/main/index.html`
- `reference/bbclient/src/app/bb-list.component.html`
- `reference/bbclient/src/app/bb-detail.component.html`

Из них берем UX-структуру и состав экранов, но не копируем реализацию.

## Базовые продуктовые сценарии

Первый frontend foundation должен поддерживать следующие сценарии:

1. Гость открывает главную страницу и видит список свежих объявлений.
2. Гость может перейти к регистрации и входу.
3. Пользователь может войти, выйти и понять, что он авторизован.
4. Пользователь может открыть объявление и увидеть его основные данные.
5. Пользователь может перейти к категориям и использовать их как способ навигации.

Сценарии управления профилем, комментариями, редактированием объявления и сложными кабинетными потоками пока не являются основой для первого дизайн-контура, но должны учитываться как будущие расширения.

## Карта экранов

### Экран 1. Главная

Назначение:

- входная точка приложения;
- список свежих объявлений;
- быстрый переход к auth и категориям.

Минимальный состав:

- верхняя навигация;
- заголовок страницы;
- список объявлений;
- блок категорий;
- действия входа или состояние текущего пользователя.

Legacy intent:

- в старом проекте главная показывала последние объявления;
- категории были постоянной частью навигации;
- auth-действия находились в верхней части интерфейса.

### Экран 2. Вход

Назначение:

- дать короткий и понятный путь к авторизации.

Минимальный состав:

- заголовок;
- форма входа;
- сообщение об ошибке;
- ссылка на регистрацию;
- понятное состояние загрузки.

### Экран 3. Регистрация

Назначение:

- создать аккаунт без перегруженного интерфейса.

Минимальный состав:

- заголовок;
- форма регистрации;
- сообщения об ошибках валидации;
- ссылка на вход;
- состояние загрузки и успешного завершения.

### Экран 4. Детальная страница объявления

Назначение:

- показать объявление как отдельную сущность;
- дать устойчивую основу для будущих действий владельца и комментариев.

Минимальный состав:

- заголовок объявления;
- изображение или заглушка;
- описание;
- цена;
- дата публикации или создания;
- категория;
- контактный или служебный блок.

Legacy intent:

- объявление состоит из содержимого, цены, даты и визуального блока;
- это отдельный экран, а не модальное окно.

### Экран 5. Базовый защищенный пользовательский контур

На первом этапе это может быть не полноценный кабинет, а минимальный экран или section, подтверждающий:

- пользователь авторизован;
- у приложения есть текущий пользователь;
- можно показать действия вроде `logout` и в будущем перейти к `my ads`.

## Общие зоны интерфейса

### 1. App Shell

Постоянный каркас приложения, который должен быть единым для первых экранов.

Состав:

- верхняя полоса навигации;
- основной контейнер контента;
- опциональная боковая зона категорий на широком экране;
- нижняя зона с минимумом служебной информации.

Требование:

- app shell должен одинаково работать для гостя и авторизованного пользователя;
- auth-страницы могут использовать упрощенный вариант shell, но не отдельный визуальный язык.

### 2. Верхняя навигация

Должна решать только основные задачи:

- переход на главную;
- доступ к категориям;
- вход и регистрация для гостя;
- текущий пользователь и выход для авторизованного пользователя.

Навигация не должна сразу пытаться вместить весь будущий кабинет.

### 3. Зона категорий

Категории в legacy были важной осью навигации. В новом приложении это стоит сохранить функционально.

На первом этапе зона категорий должна:

- поддерживать список категорий;
- быть пригодной для размещения в sidebar или как компактный блок;
- не навязывать сложную древовидную механику, пока не нужна реальная иерархия.

### 4. Основная зона контента

Это главная рабочая область, где размещаются:

- список объявлений;
- детали объявления;
- формы auth;
- будущие пользовательские действия.

Требование:

- layout должен быть устойчивым;
- ширина контента и отступы должны быть единообразны между экранами.

## Reusable UI-блоки

Ниже перечислены блоки, которые стоит спроектировать до начала активной верстки экранов.

### Примитивы

- `Button`
- `TextField`
- `Textarea`
- `FieldMessage`
- `FormField`
- `LinkAction`

### Составные блоки

- `PageContainer`
- `AppHeader`
- `CategoryList`
- `AuthForm`
- `AuthPanel`
- `AdCard`
- `AdMeta`
- `EmptyState`
- `StatusNotice`

### Почему именно они

Этот набор закрывает первый рабочий контур:

- auth;
- список объявлений;
- детальный экран;
- базовые пустые и ошибочные состояния.

Этого достаточно, чтобы не уходить в преждевременный design system ради самого design system.

## Базовые визуальные правила

Новый frontend foundation должен быть:

- спокойным;
- прикладным;
- читаемым;
- нейтральным по стилю;
- пригодным для постепенного роста.

### Что фиксируем сразу

- типографическую шкалу;
- spacing scale;
- цвета текста, поверхности, границы и состояний;
- единый небольшой радиус скругления;
- состояния `default`, `hover`, `focus`, `disabled`, `error`, `success`;
- правила ширины контейнера и вертикального ритма.

### Что не фиксируем слишком рано

- сложную брендовую графику;
- декоративные паттерны;
- тяжелую маркетинговую подачу;
- отдельный визуальный стиль для каждой страницы.

Основа должна выглядеть как каркас продукта, а не как временный лендинг.

## Контентные и UX-принципы

### Auth

- форма должна быть короткой;
- ошибки должны показываться рядом с полем или рядом с формой;
- состояния загрузки должны быть явными;
- после успешного входа пользователь должен сразу видеть, что сессия существует.

### Список объявлений

- карточка должна быстро отвечать на вопрос: что это, сколько стоит, когда создано;
- элементы списка должны сканироваться вертикально без визуального шума;
- изображение допустимо как вторичный элемент, а не как единственная точка внимания.

### Детальный экран объявления

- заголовок, цена и описание должны читаться без поиска по странице;
- служебная мета-информация должна быть визуально отделена от основного содержимого;
- блок должен быть готов к будущему расширению действиями владельца, комментариями и похожими объявлениями.

## Предлагаемая структура работы по этапам

### Этап 1. Зафиксировать foundation

Сделать:

- app shell;
- набор токенов;
- примитивы для кнопок и полей;
- `PageContainer`;
- `AppHeader`;
- `AuthPanel`.

Результат:

- можно собирать auth-страницы без копирования стилей.

### Этап 2. Собрать статические экраны

Сделать статические версии:

- главной;
- входа;
- регистрации;
- карточки объявления;
- детали объявления.

Результат:

- понятна компоновка;
- видны реальные пробелы в foundation;
- можно корректировать UI до подключения логики.

### Этап 3. Подключить первый реальный flow

Сделать:

- login;
- register;
- current user;
- logout;
- состояние сессии в shell.

Результат:

- первый полный пользовательский сценарий через весь стек.

### Этап 4. Перейти к списку категорий и созданию объявления

После auth foundation становится разумно подключать:

- список категорий;
- список объявлений;
- форму создания объявления.

## Что стоит считать Definition of Done для foundation-документа

Foundation можно считать достаточно определенным, если:

- есть понятная карта первых экранов;
- зафиксированы общие зоны интерфейса;
- определен минимальный набор reusable-блоков;
- описано, что берем из legacy, а что не переносим;
- есть порядок внедрения без попытки проектировать весь продукт сразу.

## Практическое решение для следующего шага

Следующим прикладным шагом после этого документа стоит сделать не весь дизайн сразу, а один узкий deliverable:

`app shell + auth form primitives + статический экран login`

Это даст:

- проверку выбранного визуального направления;
- проверку базовых компонентов;
- основу для реального auth flow без лишних переделок.
