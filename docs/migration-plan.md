# План миграции `bboard` на современный SPA-стек

## Новый вектор

Цель теперь не "быстро переписать Django-приложение", а собрать нормальную современную систему:

- отдельный `SPA` на `SvelteKit + TypeScript`;
- отдельный Python API-бэкенд;
- строгая типизация на фронте и бэке;
- явная доменная архитектура;
- нормальный DX: линтеры, тесты, сторибук, e2e, миграции, инфраструктура для файлов и почты.

## Что есть в старом приложении

Старый проект в `reference/bboard` это компактная доска объявлений на Django 4.1:

- регистрация, активация пользователя, логин, профиль, сброс пароля;
- двухуровневые рубрики;
- объявления с ценой, контактами и изображениями;
- комментарии;
- HTML-страницы;
- простой REST API;
- Django admin для рубрик, объявлений, комментариев и пользователей.

Отдельный Angular-клиент в `reference/bbclient` покрывает только список объявлений, карточку и комментарии.

## Что переносить нельзя как есть

- `price` сейчас хранится как `FloatField`. В новой версии нужен `Decimal/Numeric`.
- В личном кабинете нет проверки владельца объявления: чужие записи можно открыть, изменить и удалить по id.
- Автор объявления приходит через скрытое поле формы, а не назначается сервером из текущего пользователя.
- API комментариев принимает `author` из payload, то есть имя автора можно подделать.
- Тестов нет вообще.
- Старый фронтенд и API живут как два разрозненных куска без общего контракта типов.

## Целевой стек

### Backend

- `Litestar`
- `SQLAlchemy 2.x`
- `Advanced Alchemy`
- `PostgreSQL`
- `Alembic`
- `Pydantic v2`
- `pydantic-settings`
- `uv`
- `Ruff`
- `pytest`

### Frontend

- `SvelteKit`
- `TypeScript`
- `TanStack Query` для server state
- `Zod` для клиентских схем и форм
- `Tailwind CSS` + собственные design tokens
- `Storybook`
- `Vitest`
- `Playwright`
- `openapi-typescript` для генерации типов из OpenAPI

### Infra

- `Docker Compose` для локального окружения
- `PostgreSQL`
- `MinIO` или `S3`-совместимое хранилище для изображений
- `Mailpit` для локальной почты
- `Redis` опционально, если добавим очереди, кэш или rate limiting

## Почему такой стек

- `SvelteKit`, а не просто `Svelte + Vite`: есть маршрутизация, нормальная структура проекта и при этом можно делать отдельный SPA.
- `TypeScript` обязателен, иначе вы теряете половину ценности отдельного фронтенда.
- `TanStack Query` закрывает кэш, инвалидацию, загрузочные состояния и синхронизацию с API лучше, чем самописные сторы под HTTP.
- `openapi-typescript` связывает Python OpenAPI и фронтенд без ручного дублирования типов.
- `Litestar + SQLAlchemy` оставляют вам современный Python backend без завязки на Django-магии.
- `Storybook + Playwright + Vitest` дают не "игрушечный SPA", а нормальный фронтендный workflow.

## Архитектурные решения

### Общая схема

- монорепа;
- отдельное приложение API;
- отдельное приложение web;
- единый контракт через OpenAPI;
- фронтенд не знает о внутренних моделях бэка, только о DTO и API-контракте.

### Рекомендуемая структура

```text
.
├── apps/
│   ├── api/
│   └── web/
├── packages/
│   └── api-types/
├── docs/
└── infra/
```

### Что хранить на backend

- доменные модели;
- бизнес-правила;
- авторизацию и права доступа;
- загрузку и валидацию файлов;
- модерацию;
- email-уведомления;
- OpenAPI-схему.

### Что хранить на frontend

- маршруты и layout;
- UI-компоненты;
- локальное UI-state;
- запросы к API;
- optimistic updates только там, где они реально улучшают UX;
- дизайн-токены и визуальную систему.

## Выбор по auth

Для браузерного SPA я рекомендую не устраивать культ `JWT` ради слова "современно".

Базовый вариант:

- `HttpOnly` cookie;
- backend-controlled session auth;
- фронтенд ходит в API с `credentials: include`;
- API проксируется под тот же origin, чтобы не плодить CORS-ад.

Если позже появятся:

- мобильный клиент;
- внешние интеграции;
- публичный API для третьих сторон;

тогда можно добавить отдельный token-based слой.

## API-first / Contract-first

Ключевое решение: фронтенд не пишет типы руками.

Поток должен быть таким:

1. backend публикует OpenAPI;
2. frontend генерирует типы через `openapi-typescript`;
3. поверх типов строится тонкий API client;
4. `TanStack Query` работает уже поверх typed client.

Это убирает целый класс багов "на фронте думали, что поле строка, а на бэке это nullable object".

## Фронтендная часть

### Базовые принципы

- не собирать "админку из карточек и серых кнопок";
- сначала задать визуальную систему: типографика, spacing, radius, color tokens, состояния;
- все повторяющиеся компоненты сразу выносить в UI-layer;
- страницы собирать из composable блоков, а не из giant components.

### Что стоит сделать сразу

- `src/lib/ui` для примитивов и compound components;
- `src/lib/api` для typed client и query hooks;
- `src/lib/stores` только для локального client state;
- `src/routes` только для маршрутов и композиции экранов;
- `Storybook` для UI и edge-cases;
- базовую темизацию через CSS variables.

### Что не делать

- не складывать весь API в компоненты;
- не хранить server state в writable stores, если это зона `TanStack Query`;
- не делать локальный `fetch` в каждом компоненте;
- не тащить с ходу тяжелую UI-библиотеку, если можно собрать свой маленький design system.

## Backend часть

### Что поменять в домене

- `Rubric + SuperRubric + SubRubric` заменить на одну таблицу `Category` с `parent_id`;
- цену хранить как `Numeric`;
- у `Comment` сделать явную модель автора:
  - `user_id nullable`;
  - `author_name` как снапшот имени;
- права на объявление проверять на сервере, не на клиенте;
- загрузки файлов вынести в отдельный сервисный слой.

### Что поменять в архитектуре

- `api/http` для роутов и DTO;
- `domain` для сущностей и бизнес-правил;
- `services` для use cases;
- `repositories` для работы с БД;
- `integrations` для почты, файлового хранилища и фоновых задач;
- `tests` отдельно по слоям.

## Этапы работ

### Этап 0. Product freeze

- выписать все пользовательские сценарии из старого кода;
- определить, что входит в `MVP`;
- решить, нужны ли гостевые комментарии;
- решить, нужен ли этап email-активации в первой версии;
- решить, нужен ли staff backoffice сразу.

### Этап 1. Monorepo и инфраструктура

- разложить проект на `apps/api` и `apps/web`;
- поднять `docker-compose` с `postgres`, `mailpit`, `minio`;
- настроить `.env` и secrets для локальной разработки;
- настроить базовые команды запуска и тестов.

### Этап 2. Backend foundation

- поднять `Litestar`;
- настроить конфиг через `pydantic-settings`;
- подключить `SQLAlchemy` и `Alembic`;
- описать новые модели;
- настроить миграции;
- настроить OpenAPI;
- подключить линтеры и тестовую базу.

### Этап 3. Frontend foundation

- создать `SvelteKit` приложение с `TypeScript`;
- настроить alias-структуру;
- подключить `Tailwind`;
- подключить `TanStack Query`;
- подключить `Storybook`, `Vitest`, `Playwright`;
- настроить генерацию типов из OpenAPI.

### Этап 4. Auth и user flows

- регистрация;
- вход;
- выход;
- текущий пользователь;
- профиль;
- смена пароля;
- восстановление доступа;
- защита приватных маршрутов и экранов.

### Этап 5. Категории и объявления

- дерево категорий;
- список объявлений;
- фильтры;
- поиск;
- карточка объявления;
- создание и редактирование объявления;
- удаление только владельцем;
- загрузка изображений;
- галерея на фронте.

### Этап 6. Комментарии и уведомления

- список комментариев;
- добавление комментариев;
- модерация;
- email-уведомления автору объявления;
- rate limiting и антиспам, если оставляем гостей.

### Этап 7. Backoffice

- staff login;
- управление категориями;
- модерация комментариев;
- просмотр пользователей;
- технические операции без попытки клонировать весь Django admin.

### Этап 8. Полировка

- error boundaries и empty states;
- loading skeletons;
- optimistic updates там, где они реально полезны;
- observability и structured logging;
- импорт данных из старой SQLite;
- деплой и работа с файлами в production.

## Приоритет MVP

Если собирать первый релиз разумно, я бы шел так:

1. backend skeleton;
2. frontend skeleton;
3. auth;
4. категории;
5. список и карточка объявлений;
6. создание и редактирование объявлений;
7. комментарии;
8. uploads;
9. staff/backoffice;
10. email и вторичные улучшения.

## Итоговое решение

Под ваш новый запрос я бы фиксировал стек так:

- `Litestar + SQLAlchemy + PostgreSQL + Alembic` на бэке;
- `SvelteKit + TypeScript + TanStack Query + Zod + Tailwind` на фронте;
- `Storybook + Vitest + Playwright` для фронтенд-качества;
- `OpenAPI -> openapi-typescript` как связка между API и web;
- `MinIO/S3` под изображения;
- cookie/session auth для браузера вместо бессмысленного `localStorage JWT`.

Это уже не "перенос старого Django-проекта", а нормальный rewrite в современную fullstack-архитектуру.
